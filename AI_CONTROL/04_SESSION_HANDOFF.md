## Critical insights from evidence

### 1. The Height vs. Specification Problem

**Your consolidated spec:**

> "A proposed pole cannot always have a physically measured height because it does not exist yet."

**Solution:**
- Existing poles: "Measured height missing — clearance check impossible" (BLOCKER)
- Proposed poles: "Proposed pole specification required (e.g., 11m Medium Pole)" (REVIEW)
- Context records: Height field hidden (not applicable)

**This eliminates hundreds of false-positive QA errors.**

### 2. GridFlow's Additive Value (CORRECTED)

**CRITICAL CORRECTION:**

GridFlow must show records as clearly as Field Maps PLUS add design-readiness interpretation.

**The 5 layers GridFlow provides:**
1. ✅ What records are on the map (Field Maps parity)
2. ✅ What this survey data means for design
3. ✅ What is missing or incomplete
4. ✅ What remains provisional
5. ✅ What must be checked before CAD/design

**This is additive value, not replacement.**

### 3. Shape ≠ Color Paradigm

**Your exact specification:**

> "Feature type answers: what is this? QA status answers: does it need review? These should not be collapsed into the same colour or symbol."

**Shape = Asset Type:**
- Circle: Existing Pole
- Square: Proposed Pole
- Triangle: Stay/Anchor
- Diamond: Context/Crossing

**Color = Review Status:**
- Grey: Existing, passed, no action
- Blue: Proposed, ready for CAD
- Red: Critical blocker
- Orange: Review required

### 4. Proximity Is Not Proof

**Your consolidated spec:**

> "A proposed pole near an existing pole may be a replacement, a repositioned support, a nearby retained pole, a separate asset, a duplicate capture or an unrelated structure. GridFlow should use proximity as a prompt for human review, not as a final engineering conclusion."

**Safe wording prevents overclaiming.**
