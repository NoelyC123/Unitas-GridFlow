PART 1: PROJECT ORIGIN AND VISION
1.1 Why GridFlow Exists
The Problem GridFlow Solves
GridFlow was born from direct exposure to real-world friction in UK electrical distribution overhead line (OHL) survey-to-design workflows.
The problem is NOT:
	•	Bad field surveying
	•	Lack of GIS tools
	•	Poor CAD software
	•	Inadequate mobile apps
The problem IS: Survey data arrives at design teams in a weak, fragmented, unauditable form that wastes office time on clarification, interpretation, and evidence-hunting instead of actual engineering design work.
Real-World Manifestation
What happens without GridFlow:
	•	Surveyor captures data in Field Maps, Trimble controller, or notebook
	•	Raw export arrives weeks later as CSV dump (coordinates, feature codes, sparse attributes)
	•	Designer opens file and discovers:
	•	Heights missing or estimated
	•	Material/condition not captured
	•	Stay evidence absent at angle poles
	•	EX/PR replacement relationships unclear
	•	Span distances questionable
	•	Clearance data missing
	•	No provenance (who measured what, when, how accurate)
	•	Designer must:
	•	Cross-reference field notes (if they exist)
	•	Find marked-up plans (if they exist)
	•	Chase photos (if they were taken)
	•	Call surveyor (if they remember the job)
	•	Make educated guesses
	•	Flag for site revisit (costly, slow)
	•	Result: Design delayed, costs rise, office time wasted on survey archaeology
The key insight:
"The biggest inefficiency sits in the survey-to-design HANDOFF, not in the engineering design itself."
What GridFlow Does About It
GridFlow acts as the intelligent gate between survey and design.
It:
	•	Ingests raw survey exports (multiple formats, coordinate systems)
	•	Parses electrical asset schema (poles, spans, stays, cables, context, equipment)
	•	Normalizes survey connectivity (EX↔PR replacement logic, route sequencing)
	•	Assesses completeness and design-readiness
	•	Flags missing/questionable evidence
	•	Visualizes asset lifecycle states
	•	Surfaces design blockers BEFORE design time is wasted
	•	Produces design-ready artefacts (sequenced routes, QA reports, review workspaces)
The value is trust and interpretation, not just parsing:
	•	"This pole exists, height measured 9.2m by surveyor J.Smith on 2024-03-15, GNSS accuracy ±0.15m"
	•	vs. "This pole exists, height unknown — field notes required before design"
	•	vs. "This proposed pole does not exist yet — specification is a design decision"
GridFlow tells designers exactly:
	•	What IS present in the digital file
	•	What is MISSING from the digital file
	•	What that PREVENTS (design blockers)
	•	Where to look for missing evidence (field notes, photos, plans)

The Fully Realized User Experience
Surveyor (Field):
	•	Opens GridFlow mobile app on tablet
	•	Selects project/circuit
	•	Walks route, capturing poles with structured form:
	•	Tap pole location on map (GPS auto-captured)
	•	Select type (existing/proposed/angle/terminal)
	•	Measure height (laser rangefinder → auto-populate)
	•	Photo: full pole (auto-tagged, GPS-linked)
	•	Photo: pole-top (auto-tagged)
	•	Select material (dropdown: wood/concrete/steel)
	•	Rate condition (good/fair/poor/unsafe)
	•	If angle: capture stay direction, photo anchor
	•	Add remarks (voice note or text)
	•	Submit → syncs when online
	•	Continues route, GridFlow auto-sequences
	•	At end of day: reviews completeness, flags uncertainties, submits for review
Reviewer (Office, Day 1):
	•	Opens GridFlow web interface
	•	Selects submitted survey
	•	Reviews map: auto-sequenced route, lifecycle visualization, QA flags
	•	Clicks pole with warning: "Height estimated, not measured"
	•	Checks photo: can't clearly see pole-top
	•	Adds comment: "Surveyor: please revisit, measure height with rangefinder"
	•	Assigns back to surveyor
	•	Reviews angle pole: "Stay evidence not captured"
	•	Checks field notes: no mention of stay
	•	Checks photos: no anchor visible
	•	Flags: "Site revisit required - stay configuration unknown"
	•	Reviews EX↔PR pairings: 15 suggested matches
	•	Confirms 12 (proximity + logic clear)
	•	Rejects 2 (wrong poles paired)
	•	Leaves 1 uncertain (designer decision)
	•	Approves survey for design (with flagged items)
Designer (Office, Day 2-5):
	•	Opens GridFlow design workspace
	•	Imports survey (sequenced route auto-loaded)
	•	Reviews issue register:
	•	3 poles require site revisit (heights/stays missing)
	•	5 poles need specification (proposed poles)
	•	2 span clearances need verification (road crossings)
	•	Filters map: "Design blockers only"
	•	Sees 3 critical issues
	•	Adds to site visit request
	•	Begins design on ready sections:
	•	Pole specifications (uses measured heights, conditions, loadings)
	•	Conductor selection (uses voltage, span distances, clearance requirements)
	•	Stay design (uses captured angles, existing stay evidence)
	•	Material optimization (uses condition ratings, replacement priorities)
	•	Exports to PoleCAD: ready to design (no survey archaeology needed)
	•	Completes design in 60% less time (no clarification delays)
DNO Approver (Office, Day 6):
	•	Reviews completed design
	•	Checks GridFlow evidence pack:
	•	Survey completeness: 95% (5% flagged for revisit)
	•	Design-readiness: READY (all blockers resolved)
	•	QA status: 2 warnings (acceptable), 0 failures
	•	Evidence quality: HIGH (measured data, photos, GNSS <0.2m)
	•	Approves design for construction
	•	GridFlow updates asset register (as-designed records)
Construction Team (Site, Weeks 1-4):
	•	Accesses GridFlow mobile (construction mode)
	•	Views design route on map
	•	At each pole:
	•	Sees design specification
	•	Compares to as-found (GridFlow survey data visible)
	•	Records as-built (actual installed spec)
	•	Photos installed pole
	•	Submits as-built records
	•	GridFlow updates asset register (final as-built)

   - qa_engine.py (run validation checks)
   - Generate outputs:
     * meta.json (summary, completeness, design-readiness)

     * sequenced_route.json (design chain)
3. View Map → map_preview.py:
   - Load map_data.json, sequenced_route.json
   - Runtime enrichment (lifecycle states, popup data, span features)
   - Render map_viewer.html
   - map-viewer.js handles UI (filters, popups, lifecycle viz)
4. Download PDF → pdf_reports.py (ReportLab generation)
5. Browse Jobs → jobs.py (list all processed jobs)

Coordinate System Handling
Supported Systems:
	•	OSGB36 (British National Grid, EPSG:27700)
	•	England, Scotland, Wales
	•	Easting/Northing in meters
	•	Origin: Ordnance Survey datum
	•	Irish Grid TM65 (EPSG:29902)
	•	Northern Ireland, Republic of Ireland
	•	Easting/Northing in meters
	•	Origin: Geodetic Survey of Ireland
	•	WGS84 (GPS, EPSG:4326)
	•	Global latitude/longitude
	•	Converted on intake to regional grid
Conversion Pipeline:
python
# pyproj handles all transformations
from pyproj import Transformer

# Example: Irish Grid → WGS84 for Leaflet display
transformer = Transformer.from_crs("EPSG:29902", "EPSG:4326", always_xy=True)
lat, lng = transformer.transform(easting, northing)
Detection Logic:
	•	Easting 0-700000, Northing 0-1300000 → OSGB
	•	Easting 0-500000, Northing 0-500000 → Irish Grid
	•	Lat/Lng → WGS84

2.2 Phase Development History
Phase 0: Pre-Project (2023-2024)
Real-world observations:
	•	Working on UK OHL refurbishment projects
	•	Seeing survey-to-design handoff friction repeatedly
	•	Designers spending 40-60% of time on "survey archaeology"
	•	Missing heights, stays, materials forcing site revisits
	•	Weak controller exports requiring manual interpretation
	•	No structured gate between survey and design
Key insight: The problem is not individual tools (Field Maps works, PoleCAD works), but the handoff gap between them.

Phase 1: MVP Build (Early 2025)
Goal: Prove basic intake → QA → output flow works
Delivered:
	•	Flask app structure (blueprint-based routes)
	•	CSV upload endpoint
	•	Basic controller parsing (Trimble format)
	•	Placeholder QA rules (coordinate bounds, duplicate IDs)
	•	Map viewer (Leaflet, basic point display)
	•	PDF report generation (ReportLab)
	•	Job storage (filesystem-based)
Test Count: ~15-20 tests
Validation: Synthetic test files only

2.3 Key Technical Decisions and Trade-offs
Decision 1: Filesystem Storage vs. Database
Chosen: Filesystem-based job storage
Why:
	•	Simple deployment (no DB setup)
	•	Easy backup/migration (copy folder)
	•	Direct file access (no ORM complexity)
	•	Human-readable outputs (JSON, CSV)
	•	Sufficient for current scale (100s of jobs, not 1000s)
Trade-off:
	•	Harder to query/search across all jobs
	•	No ACID transactions
	•	Scaling ceiling at ~10,000 jobs
Future: Will need database for multi-user Stage 6+ (PostgreSQL/PostGIS likely)

Decision 2: Runtime Enrichment vs. Full Persistence
Chosen: Hybrid — basic attributes in map_data.json, enrichment in map route
Why:
	•	Lighter stored JSON (smaller files, faster upload)
	•	Flexibility (can change enrichment logic without re-processing)
	•	Separation of concerns (intake vs. display)
Trade-off:
	•	Map route is "source of truth" for lifecycle/popup data (not stored JSON)
	•	Downstream tools can't rely on stored JSON alone
	•	Requires understanding of enrichment pipeline
Future: Stage 4+ will need full persistence (API access, offline use)

Decision 3: Coordinate System Auto-detection vs. User Selection
Chosen: Auto-detection based on coordinate ranges
Why:
	•	Better UX (no manual selection)
	•	Works for 95% of files (OSGB, Irish Grid ranges don't overlap)
	•	Fallback to WGS84 if ambiguous
Trade-off:
	•	Can misdetect if coordinates are edge-of-range
	•	No override mechanism (yet)
Future: Stage 4 will add manual override + validation warning

Decision 4: Monolithic map-viewer.js vs. Modular Components
Chosen: Monolithic (~2,809 lines)
Why:
	•	Faster initial development (no build tooling)
	•	Vanilla JS (no framework lock-in)
	•	All logic in one place (easier to understand)
Trade-off:
	•	Large file (hard to navigate)
	•	Risky for AI-driven edits (fragile)
	•	No component reuse
Future: Stage 5 will need refactor (Vue/React components, build tooling)

Decision 5: Pytest vs. Other Test Frameworks
Chosen: pytest
Why:
	•	Python standard (widely used)
	•	Simple syntax (no boilerplate)
	•	Good fixtures/parametrization
	•	Coverage integration
Trade-off:
	•	No browser automation (Playwright/Selenium separate)
	•	Backend-focused (frontend tests manual)
Future: Stage 5 will add Playwright for UI testing

Asset Types (Electrical Schema)
1. Poles/Support Structures
Types:
	•	Existing (EX) — currently in field, may be retained/replaced/recovered
	•	Proposed (PR/Pol) — new pole to be installed
	•	Angle — route deviation >10°, requires stays
	•	Terminal — end of line, requires terminal stay
	•	Intermediate — straight-through, no angle
	•	Tee-off — branch point, requires tee-off stay
Materials:
	•	Wood (most common, older networks)
	•	Concrete (modern standard)
	•	Steel (high-load, special applications)
	•	Composite (modern, lightweight)
Condition:
	•	Good (no defects, full service life remaining)
	•	Fair (minor defects, serviceable, monitor)
	•	Poor (significant defects, near end-of-life, plan replacement)
	•	Unsafe (structural failure risk, immediate action required)
Heights:
	•	7-12m typical for 11kV distribution
	•	Measured vs. estimated (measured = better)
	•	Missing height = design blocker (can't calculate clearances)

2. Spans (Overhead Line Connections)
Typical Lengths:
	•	11kV: 40-100m normal, 100-300m extended, >300m rare (probable missing pole)
	•	33kV: 60-200m normal, >500m rare
	•	132kV: 100-400m normal
Anomalies:
	•	<10m: Probable GPS error or duplicate capture
	•	500m (11kV): Probable missing intermediate pole
	•	Inconsistent bearing: Route deviation, check for angle pole
Design Factors:
	•	Conductor sag (longer span = more sag)
	•	Clearance (ground, road, buildings, trees)
	•	Loading (ice, wind, tension)

3. Stays/Anchors (Mechanical Support)
When Required:
	•	Angle poles (>10° deviation)
	•	Terminal poles (end of line)
	•	Tee-offs (branch points)
	•	High-load poles (transformers, heavy conductors)
Types:
	•	Angle stay — opposes line tension change at angle
	•	Terminal stay — opposes line tension at end
	•	Tee-off stay — opposes branch tension
	•	Tandem stay — multiple stays in line for extra support
Anchor Types:
	•	Screw anchor (most common, augers into ground)
	•	Block anchor (concrete block buried)
	•	Deadman anchor (timber beam buried)
	•	Rock anchor (drilled into rock)
Stay Insulators:
	•	Required to break electrical path to ground
	•	Missing insulator = electrical hazard
Missing Stay Evidence = Design Blocker:
	•	Can't design pole without knowing stay configuration
	•	Can't calculate loading without stay support
	•	Site revisit required if no field evidence (notes/photos)

4. Cables (Underground/Aerial)
Types:
	•	Underground cable (UG, buried)
	•	Aerial bundled conductor (ABC, overhead, insulated)
	•	Service drops (pole to customer)
Relevance to GridFlow:
	•	Often captured as context features
	•	Less design-critical than poles/spans
	•	Future Stage 4+ feature (cable-pole transitions)

5. Context Features (Non-Electrical Assets)
Types:
	•	Roads (crossing clearance critical)
	•	Railways (DNO/Network Rail coordination required)
	•	Rivers/canals (crossing permits)
	•	Fences/gates (access constraints)
	•	Buildings (clearance, wayleaves)
	•	Trees (vegetation management)
	•	Other utilities (BT, gas, water, telecoms)
Crossing Priorities:
	•	CRITICAL: 110kV/132kV power lines, railways, motorways
	•	HIGH: Roads, 33kV lines, rivers
	•	MEDIUM: BT/telecoms, 11kV lines, minor roads
	•	LOW: Fences, gates, field boundaries
Why Context Matters:
	•	Clearance requirements (statutory minimums)
	•	Access planning (how to reach poles)
	•	Wayleave permissions (landowner consent)
	•	Construction constraints (traffic management, permits)

6. Equipment (Transformers, Switchgear)
Types:
	•	Transformer (voltage conversion, 11kV → 400V)
	•	Switch (circuit isolation, reconfiguration)
	•	Fuse (overcurrent protection)
	•	Recloser (automatic fault clearing)
	•	Sectionalizer (fault location)

	•	Requires larger pole, additional stays

def detect_coordinate_system(df):
    """
    Auto-detect based on coordinate ranges:
    - OSGB: E: 0-700K, N: 0-1.3M
    - Irish Grid: E: 0-500K, N: 0-500K
    - WGS84: Lat: -90 to 90, Lng: -180 to 180
    """
    if 'easting' in df.columns:
        e_min, e_max = df['easting'].min(), df['easting'].max()
        n_min, n_max = df['northing'].min(), df['northing'].max()

        if e_max < 500000 and n_max < 500000:
            return 'Irish_Grid_TM65'
        elif e_max < 700000 and n_max < 1300000:
            return 'OSGB36'

    return 'WGS84'

    code = row.get('feature_code', '').lower()
    function = row.get('function', '').lower()

        if any(c.lower() in code or c.lower() in function for c in codes):

                'primary_type': family,
                'record_type': code,
                'confidence': 'HIGH'
            }

    # Fallback to context
    return {
        'primary_type': 'context_feature',
        'record_type': 'unknown',
        'confidence': 'LOW'
    }

def detect_ex_pr_pairs(records, max_distance=50):
    """
    Find probable EX↔PR replacement pairs.
    Returns list of matches with distance, status.
    """
    ex_poles = [r for r in records if 'ex' in r['record_type'].lower()]
    pr_poles = [r for r in records if 'pr' in r['record_type'].lower() or 'pol' in r['record_type'].lower()]

    matches = []
    for ex in ex_poles:
        for pr in pr_poles:
            dist = calculate_distance(ex, pr)

            if dist < max_distance:
                matches.append({
                    'ex_pole': ex['point_id'],
                    'pr_pole': pr['point_id'],
                    'distance': dist,
                    'status': 'suggested' if dist < 50 else 'uncertain',
                    'reviewed': False
                })

    return matches

# route_sequencer.py
def sequence_route(poles):
    """
    Auto-sequence poles into design chain.
    Algorithm: nearest-neighbor with bearing constraints.
    """
    if not poles:
        return []

    # Start at lowest point_id or explicit start marker
    chain = [poles[0]]
    remaining = poles[1:]

    while remaining:
        current = chain[-1]

        # Find nearest pole within bearing tolerance
        candidates = []
        for pole in remaining:
            dist = calculate_distance(current, pole)
            bearing = calculate_bearing(current, pole)

            # Check bearing consistency (if not first span)
            if len(chain) > 1:
                prev_bearing = calculate_bearing(chain[-2], current)
                bearing_diff = abs(bearing - prev_bearing)
                if bearing_diff > 180:
                    bearing_diff = 360 - bearing_diff

                # Prefer bearing consistency (<30° deviation)
                confidence = 'HIGH' if bearing_diff < 30 else 'MEDIUM'
            else:
                confidence = 'HIGH'

            candidates.append({
                'pole': pole,
                'distance': dist,
                'bearing': bearing,
                'confidence': confidence
            })

        # Sort by distance, pick nearest
        candidates.sort(key=lambda x: x['distance'])
        next_pole = candidates[0]

        chain.append({
            **next_pole['pole'],
            'sequence_position': len(chain) + 1,
            'distance_from_prev': next_pole['distance'],
            'bearing_from_prev': next_pole['bearing'],
            'confidence': next_pole['confidence']
        })

        remaining.remove(next_pole['pole'])

    return chain

            self.check_coordinate_bounds,
            self.check_duplicate_coordinates,
            self.check_height_validity,
            self.check_span_distances,
            self.check_angle_stay_evidence,
            self.check_material_completeness,
            self.check_ex_pr_clarity,
            self.check_clearance_proximity,
            self.check_feature_code_validity,
            self.check_sequence_confidence
        ]

    def run_all_checks(self, records):
        """
        Run all QA checks on records.
        Returns issues list.
        """
        issues = []

        for check in self.checks:
            check_issues = check(records)
            issues.extend(check_issues)

        return issues

    def check_angle_stay_evidence(self, records):
        """
        Example check: angle poles should have stay evidence.
        """
        issues = []

        angle_poles = [r for r in records if 'angle' in r.get('function', '').lower()]

        for pole in angle_poles:
            # Find stay records within 20m
            nearby_stays = self._find_nearby_stays(pole, records, radius=20)

            if not nearby_stays:
                issues.append({
                    'point_id': pole['point_id'],
                    'severity': 'BLOCKER',
                    'code': 'ANGLE_NO_STAY',
                    'category': 'mechanical',
                    'message': f"Angle pole {pole['point_id']} has no stay evidence captured",
                    'recommendation': "Check field notes/photos for stay configuration. Site revisit may be required."
                })

        return issues

// map-viewer.js
function classifyLifecycleState(record, nearbyRecords) {
    const recordType = record.record_type.toLowerCase();
    const remarks = record.remarks?.toLowerCase() || '';

    // Context records
    if (recordType === 'context') return 'context';

    // Existing poles
    if (recordType.includes('ex')) {
        if (remarks.includes('retain') || remarks.includes('keep')) {
            return 'retained';
        }
        if (remarks.includes('recover') || remarks.includes('remove')) {
            return 'recovered';
        }

        // Check for nearby proposed
        const nearbyProposed = nearbyRecords.filter(r =>
            r.record_type.toLowerCase().includes('pr') ||
            r.record_type.toLowerCase().includes('pol')
        );

        if (nearbyProposed.length > 0) {
            const closest = findClosest(record, nearbyProposed);
            const dist = distance(record, closest);

            if (dist < 5) return 'repositioned';
            if (dist < 50) return 'replacement';  // or 'suggested'/'confirmed'
            return 'unmatched_ex';
        }

        return 'existing';
    }

    // Proposed poles
    if (recordType.includes('pr') || recordType.includes('pol')) {
        const nearbyExisting = nearbyRecords.filter(r =>
            r.record_type.toLowerCase().includes('ex')
        );

        if (nearbyExisting.length > 0) {
            const closest = findClosest(record, nearbyExisting);
            const dist = distance(record, closest);

            if (dist < 50) return 'replacement';
            return 'unmatched_pr';
        }

        return 'proposed';
    }

    return 'existing';  // default
}

# qa_engine.py (backend)
def detect_angle_poles(records):
    """
    Identify poles requiring stay evidence.
    """
    angle_poles = []

    for i, record in enumerate(records):
        is_angle = False
        deviation = None

        # Check explicit function
        if 'angle' in record.get('function', '').lower():
            is_angle = True

        # Check route deviation (bearing change)
        if i > 0 and i < len(records) - 1:
            prev_bearing = calculate_bearing(records[i-1], record)
            next_bearing = calculate_bearing(record, records[i+1])
            deviation = abs(next_bearing - prev_bearing)

            if deviation > 180:
                deviation = 360 - deviation

            if deviation > 10:  # >10° = angle pole
                is_angle = True

        if is_angle:
            angle_poles.append({
                'point_id': record['point_id'],
                'deviation': deviation,
                'explicit': 'angle' in record.get('function', '').lower()
            })

    return angle_poles

def scan_for_stays(angle_pole, all_records, radius=20):
    """
    Find stay records near angle pole.
    """
    nearby_stays = []

    for record in all_records:
        record_type = record.get('record_type', '').lower()

        if 'stay' in record_type or 'anchor' in record_type:
            dist = distance(angle_pole, record)

            if dist <= radius:
                nearby_stays.append({
                    'point_id': record['point_id'],
                    'type': classify_stay_type(record),
                    'distance': dist,
                    'bearing': record.get('bearing')
                })

    return {
        'angle_pole_id': angle_pole['point_id'],
        'stays_found': len(nearby_stays),
        'stay_details': nearby_stays,
        'has_stay_evidence': len(nearby_stays) > 0
    }

# span_generator.py
def calculate_span_distance(pole1, pole2):
    """
    Calculate 3D distance including elevation.
    """
    # Horizontal distance
    dx = pole2['easting'] - pole1['easting']
    dy = pole2['northing'] - pole1['northing']
    horizontal = math.sqrt(dx**2 + dy**2)

    # Vertical distance
    if 'elevation' in pole1 and 'elevation' in pole2:
        dz = pole2['elevation'] - pole1['elevation']
        return math.sqrt(horizontal**2 + dz**2)

    return horizontal

def classify_span_status(distance, voltage=11000):
    """
    Classify span based on distance and voltage.
    """
    if distance < 10:
        return {
            'class': 'duplicate',
            'severity': 'WARNING',
            'message': '⚠️ Probable duplicate pole or GPS error',
            'action': 'Review coordinates and field notes'
        }

    if voltage == 11000:  # 11kV
        if distance > 500:
            return {
                'class': 'critical_gap',
                'severity': 'BLOCKER',
                'message': '🚨 Probable missing intermediate pole',
                'action': 'Verify route and check for missing records'
            }
        elif distance > 300:
            return {
                'class': 'very_long',
                'severity': 'WARNING',
                'message': '⚠️ Very long span — verify no missing pole',
                'action': 'Check field notes and survey evidence'
            }
        elif distance > 100:
            return {
                'class': 'extended',
                'severity': 'INFO',
                'message': 'ℹ️ Long span — verify conductor sag compliance',
                'action': 'Check clearance calculations'
            }

    return {
        'class': 'normal',
        'severity': 'PASS',
        'message': '✓ Typical span'
    }
