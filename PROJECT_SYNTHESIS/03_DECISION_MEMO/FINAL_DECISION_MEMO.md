# SpanCore / EW Design Tool — Final Decision Memo

Prepared: 2026-04-18  
Status: Final decision memo derived from the restored AI source set, comparison matrix, claims ledger, and master synthesis  
Purpose: To state clearly what has been decided about SpanCore, what it is, what it is not, and what happens next

---

## 1. Decision summary

The decision is to **continue SpanCore**, but only in a **much narrower, more disciplined, proof-of-value-driven form**.

SpanCore should **not** currently be treated as:
- a broad software platform
- a general utility SaaS
- a CAD/GIS replacement
- a full engineering system
- or an AI-first product

It should instead be treated as:

**a narrow pre-CAD QA / compliance / submission-readiness tool for electricity survey-to-design handoffs**

Short version:

**SpanCore is a good narrow opportunity, not a good broad opportunity.**

---

## 2. What has been decided

### Continue the project
The project is worth continuing because:
- the workflow pain appears real
- the codebase appears recoverable
- the narrow product identity is strong
- and the project is more than just an idea

### Do not broaden the scope
The project should not expand into:
- broad multi-DNO ambitions
- enterprise platform thinking
- major architecture overbuild
- CAD/GIS replacement ambitions
- AI-heavy features
- or general smart design platform positioning

### Recover the narrow MVP first
The immediate priority is to recover one narrow end-to-end workflow so the product can actually be used and shown.

### Validate with a real user quickly
The project should not continue in isolation for long. It must be tested with:
- one real workflow
- one real user or buyer champion
- and one measurable proof-of-value

### Use consultancy leverage / internal use as the first commercial form
The most realistic early route is:
- internal operational use
- consultancy leverage
- then niche licensing only if repeatability is proven

---

## 3. What SpanCore is now

SpanCore is currently best described as:

- a meaningful technical scaffold
- a partially realised narrow product concept
- a recoverable codebase
- and a project with real potential if narrowly executed

It is **not yet**:
- a trusted product
- a broadly validated tool
- a proven commercial software business
- or a finished end-to-end system

---

## 4. What SpanCore should become

The best realistic target is:

**A narrow pre-CAD QA / compliance / submission-readiness tool for electricity survey-to-design workflows, starting with one DNO, one input path, and one proof-of-value workflow.**

Commercially, the best sequence is:

1. **internal / consultancy leverage**
2. **licensed niche product**
3. only later consider broader software ambitions

This means the project should be built around:
- one ugly recurring workflow
- one clear set of rules
- one measurable outcome
- one buyer pain worth paying to remove

---

## 5. What SpanCore should not become yet

SpanCore should not currently try to become:

- a broad engineering platform
- a multi-tenant enterprise SaaS
- a general compliance platform across many domains
- a CAD/GIS replacement
- a digital twin system
- a fully AI-driven design product
- a workflow-suite product with heavy auth / infra / permissions complexity

These are all scope traps at the current stage.

---

## 6. Core decision logic

This decision is based on five main conclusions:

1. **The narrow workflow problem appears real**
2. **The narrow product identity is much stronger than the broad one**
3. **The codebase appears worth recovering rather than rebuilding**
4. **Commercial value is plausible in a niche, but not yet proven**
5. **The project’s future should be decided by proof-of-value, not by ambition**

---

## 7. What must happen next

The next phase is not more open-ended exploration.

The next phase is:

1. recover the narrow MVP
2. define the exact proof-of-value workflow
3. validate it with one real user / team / buyer
4. measure one meaningful ROI result
5. decide the future from evidence

---

## 8. Go / pause / stop rule

### Go
Continue actively if:
- the narrow MVP becomes runnable end-to-end
- a real workflow can be completed
- and one real user confirms that it removes meaningful pain

### Pause
Pause broader development if:
- the MVP remains incomplete
- buyer/workflow clarity is still weak
- or the value case remains conceptual rather than measured

### Stop productisation
Stop treating SpanCore as a product opportunity if:
- no proof-of-value workflow is achieved
- no real buyer/user signal appears
- or the software only works when surrounded by bespoke founder knowledge

In that case, it may still survive as a personal/internal tool, but not as a product business.

---

## 9. Final statement

SpanCore is worth continuing.

But it is only worth continuing under strict discipline:
- narrow scope
- fast recovery
- real user validation
- measurable proof
- and no drift into broad product fantasy

The strongest final position is:

**Continue — but only in narrowed form, and only under proof-of-value logic.**
