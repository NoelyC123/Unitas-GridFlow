# SpanCore / EW Design Tool — Execution Alignment Plan

Prepared: 2026-04-18
Status: Immediate execution plan following the restored AI source set, comparison matrix, claims ledger, master synthesis, and final decision memo
Purpose: To convert the synthesis into a practical sequence of actions

---

## 1. Objective of this plan

This plan exists to ensure that SpanCore moves from:
- promising scaffold

to:
- narrow proof-of-value tool

without drifting back into:
- overbuilding
- platform ambition
- or open-ended planning

The principle is simple:

**Recover one narrow workflow.
Prove one real result.
Validate with one real user.
Then decide what happens next.**

---

## 2. Current working assumptions

This plan assumes:

- the codebase is worth recovering
- the narrow product identity is correct
- internal / consultancy leverage is the strongest early commercial form
- broad productisation is premature
- and proof-of-value is the real next gate

---

## 3. Phase 1 — Immediate technical recovery

### Goal
Make the narrow MVP runnable end-to-end.

### Focus
Recover the exact workflow that matters most.

### Scope
Only work on what is needed for:
- one upload path
- one QA path
- one output/reporting path
- one demonstrable workflow

### Explicitly allowed
- fixing broken or missing routes
- restoring the narrow operational flow
- getting one stable end-to-end demo working
- tightening repo clarity where needed for MVP execution

### Explicitly not allowed
- broad new feature work
- multi-DNO expansion
- heavy infrastructure work
- auth / tenancy / enterprise admin features
- broad UI redesign
- AI additions
- speculative integrations

### Output of Phase 1
A runnable narrow MVP that can be shown to a real user.

---

## 4. Phase 2 — Workflow definition

### Goal
Define one exact proof-of-value workflow.

### Must be specified clearly
- one target user
- one target buyer or workflow owner
- one input type
- one output type
- one rules source
- one definition of success
- one measurable ROI metric

### Questions that must be answered
- What exact workflow are we proving first?
- What is the ugly recurring task?
- What pain does the user feel today?
- How is it done currently?
- What will count as a better result?

### Output of Phase 2
A single written proof-of-value workflow definition.

---

## 5. Phase 3 — Real user validation

### Goal
Put the narrow MVP in front of one real user / team / buyer champion.

### Validation target
One real workflow conversation or demo with a person who actually understands the job.

### What we need from them
- Is the workflow pain real?
- Is the current product framing correct?
- Would they use this?
- Would this save them time, reduce errors, or reduce rework?
- What would stop them from using it?
- What output matters most to them?

### Output of Phase 3
A written validation note containing:
- who the user was
- what they said
- what they cared about
- what they rejected
- and what the product must change, if anything

---

## 6. Phase 4 — Proof-of-value measurement

### Goal
Measure whether SpanCore produces a meaningful result on one workflow.

### Proof metrics could include
- hours saved
- fewer manual checks
- fewer errors
- faster readiness for design/submission
- reduced rework
- reduced need for senior manual review
- clearer auditability

### Key rule
Do not rely on “this seems useful.”

Require:
- one measurable comparison
- before vs after
- current process vs SpanCore-supported process

### Output of Phase 4
One written ROI / proof-of-value note.

---

## 7. Phase 5 — Decision gate

After technical recovery and validation, make one explicit decision.

### Go
Continue active product development if:
- the MVP works
- the workflow is real
- the user signal is positive
- and the ROI case is believable

### Pause
Pause and reassess if:
- the MVP works but user demand is unclear
- the workflow is too bespoke
- the buyer path is weak
- or the ROI is too soft

### Stop productisation
Stop treating SpanCore as a product opportunity if:
- proof-of-value does not emerge
- adoption friction is too high
- the workflow does not generalise
- or the software only works through bespoke founder effort

---

## 8. Work rules going forward

### Rule 1
No major new features before proof-of-value.

### Rule 2
No broad platform language in planning documents until the MVP earns it.

### Rule 3
No commercial optimism without user or buyer evidence.

### Rule 4
No governance drift — canonical current truth must stay separate from history and aspiration.

### Rule 5
Every major next step must answer:
- does this help the narrow MVP?
- does this help proof-of-value?
- does this help user validation?

If not, it is probably not a priority.

---

## 9. Immediate priorities list

### Priority 1
Finish the narrow MVP recovery

### Priority 2
Define the first proof-of-value workflow

### Priority 3
Identify one real user / design team / buyer champion

### Priority 4
Run one real validation conversation or demo

### Priority 5
Write one proof-of-value note

---

## 10. Things we are deliberately not doing now

We are not currently:
- building a broad SaaS
- expanding to many DNOs
- replacing CAD/GIS tools
- adding AI-first features
- building enterprise auth / tenancy
- designing for scale before proof
- making marketing-led product claims
- treating roadmap ambition as current capability

---

## 11. Final execution instruction

The next stage of SpanCore is not:
- more abstraction
- more open-ended idea generation
- or more broad comparison work

It is:

**recover, validate, measure, decide**

That sequence should govern all work from here.
